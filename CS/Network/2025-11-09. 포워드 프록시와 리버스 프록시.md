### 1\. 개요

프록시(Proxy) 서버는 **클라이언트**와 **서버** 사이의 중계자 역할을 수행하는 서버다. 요청과 응답이 프록시를 통해 전달되며, 이 위치와 목적에 따라 **포워드 프록시**와 **리버스 프록시**로 구분된다.

### 2\. 포워드 프록시(Forward Proxy)

#### 2.1. 개념

> 포워드 프록시는 클라이언트와 인터넷 서버 사이에서 **클라이언트 측을 대리**하여 요청을 전달하는 서버다. 즉, 클라이언트가 외부 서버에 접근할 때 반드시 포워드 프록시를 거치는 구조다.

클라이언트는 자신이 요청하는 실제 서버의 주소를 직접 알 필요가 없다.

#### 2.2. 주요 특징 및 장점

포워드 프록시는 주로 클라이언트 측의 접근 제어, **캐싱**, 익명 접속 등을 위해 사용된다.

* **클라이언트 보안 및 접근 제어:**
    * 포워드 프록시는 특정 웹사이트(서버)로의 접근을 차단하는 필터링 역할을 수행할 수 있다.
    * 이는 주로 회사나 학교 내부망에서 인터넷 사용을 제어하는 용도로 활용된다.
    * 클라이언트가 특정 서버에 직접 접근하는 것을 방지하여 내부 네트워크를 보호한다.
* **캐싱(Caching):**
    * 클라이언트가 특정 웹페이지에 접근할 때, 프록시 서버는 해당 페이지의 데이터를 **캐싱**해 둘 수 있다.
    * 이후 동일한 페이지에 다른 클라이언트가 접근할 경우, 원본 서버에 다시 요청하지 않고 캐싱된 정보를 즉시 반환하여 응답 속도를 향상시킨다.
* **암호화 및 익명성:**
    * 클라이언트의 요청은 포워드 프록시 서버에서 암호화될 수 있다.
    * 암호화된 요청은 외부 서버를 통과할 때 최소한의 정보만 포함하므로, **클라이언트의 IP 주소**를 감추는 보안 효과를 제공한다.

### 3\. 리버스 프록시(Reverse Proxy)

#### 3.1. 개념

> 리버스 프록시는 **서버 앞단**에서 클라이언트의 요청을 받아, 내부의 실제 서버(WAS 등)로 전달하고 그 응답을 다시 클라이언트에게 반환하는 서버다. **서버 입장에서 트래픽을 대리 처리**하는 구조다.

일반적으로 **WEB(Nginx, Apache) 서버**가 **WAS(Tomcat)** 앞단에서 요청을 처리하는 구조가 리버스 프록시 형태에 해당하며, 이때 WEB 서버가 리버스 프록시 서버 역할을 수행한다.

#### 3.2. 주요 장점

* **로드 밸런싱(Load Balancing):**
    * 여러 대의 백엔드 서버 앞단에 리버스 프록시 서버를 배치하여, 특정 서버에 과부하가 걸리지 않도록 트래픽을 효율적으로 분산한다.
* **서버 보안:**
    * 클라이언트가 실제 백엔드 서버의 IP 주소나 구조를 알 수 없도록 정보를 숨긴다.
    * 이는 외부 공격으로부터 내부 서버를 보호하는 방화벽 역할(WAF)을 겸할 수 있다.
* **캐싱(Caching):**
    * 자주 요청되는 정적 콘텐츠(Static Contents)를 리버스 프록시 서버에 캐싱한다.
    * 지리적으로 클라이언트와 가까운 위치(Edge)의 리버스 프록시가 캐싱된 데이터를 반환함으로써, 원본 서버(Origin)까지 통신하지 않고도 빠른 응답을 제공한다.
* **SSL/TLS 암호화 (SSL Termination):**
    * 클라이언트와의 **SSL/TLS 암호화 통신**을 리버스 프록시 서버가 전담하여 처리할 수 있다.
    * 이를 통해 백엔드 서버는 암호화/복호화 연산 부담에서 벗어나 비즈니스 로직 처리에만 집중할 수 있다.

### 5\. 요약

**포워드 프록시**는 **클라이언트**를 대리하여 주로 내부망의 보안과 캐싱을 담당한다. 반면 **리버스 프록시**는 **서버**를 대리하여 외부 요청을 받아 로드 밸런싱, 서버 보안, 암호화, 캐싱 등을 수행한다. 두 방식 모두 네트워크 통신을 중계하며 효율성과 보안을 강화하는 핵심 기술이다.
