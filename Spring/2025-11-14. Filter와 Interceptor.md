### 1\. 개요

필터(Filter)와 인터셉터(Interceptor)는 모두 클라이언트의 요청과 서버 응답 사이에서 동작하는 **가로채기(Pre/Post-processing) 매커니즘**이다. 둘 다 공통적인 전처리 및 후처리 작업을 수행하지만, 동작하는 위치(Scope), 기반 기술, 적용 범위에서 명확한 차이가 있다.

-----

### 2\. 필터 (Filter)

#### 2.1. 정의 및 동작 위치

필터는 **서블릿 컨테이너(Servlet Container) 레벨**에서 동작한다. 스프링의 **DispatcherServlet**으로 요청이 전달되기 *전에* 실행된다.

#### 2.2. 핵심 특징

* **기반 기술:** 자바 서블릿 API (`javax.servlet.Filter`)를 기반으로 한다.
* **적용 범위:** 스프링 프레임워크와 무관하게 **애플리케이션 전체**의 요청 및 응답에 관여한다.
* **제한 사항:** 스프링 컨텍스트 외부에서 동작하므로, 스프링 빈(Bean)을 직접 주입받을 수 없다. (단, `DelegatingFilterProxy`를 통해 우회 가능하다.)
* **기능:** `ServletRequest`와 `ServletResponse` 객체를 다른 객체로 교체할 수 있어 기능적으로 더 강력하다.

#### 2.3. 주요 목적

* 요청/응답의 공통 인코딩 설정 (예: UTF-8)
* CORS(Cross-Origin Resource Sharing) 처리
* 보안 관련 헤더 추가
* XSS(Cross-Site Scripting) 또는 SQL Injection 방지 등 보안 필터링
* 전역적인 로깅 및 트래픽 모니터링

#### 2.4. 예시 코드

```java
@Component
public class MyFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, 
                         ServletResponse response, 
                         FilterChain chain) throws IOException, ServletException {
        System.out.println("필터 실행 전");
        chain.doFilter(request, response); // 다음 필터 or DispatcherServlet으로 진행
        System.out.println("필터 실행 후");
    }
}
```

### 3\. 인터셉터 (Interceptor)

#### 3.1. 정의 및 동작 위치

인터셉터는 **Spring MVC** 스택 내에서 동작한다. **DispatcherServlet**이 요청을 받은 후, \*\*컨트롤러(Handler)\*\*를 호출하기 *전과 후*에 실행된다.

#### 3.2. 핵심 특징

* **기반 기술:** 스프링이 제공하는 `HandlerInterceptor` 인터페이스를 기반으로 한다.
* **적용 범위:** **Spring MVC의 핸들러(컨트롤러) 요청 및 응답**에만 관여한다.
* **장점:** 스프링 컨테이너 내에서 동작하므로, 서비스나 리포지토리 등 **스프링 빈(Bean)을 자유롭게 주입**받을 수 있다.
* **정교한 제어:** 실행 시점을 컨트롤러 실행 전(`preHandle`), 컨트롤러 실행 후(`postHandle`), 뷰 렌더링 후(`afterCompletion`)로 세분화하여 정교한 제어가 가능하다.

#### 3.3. 주요 목적

* **인증 및 인가:** 특정 페이지 접근 전, 사용자의 로그인 여부나 권한 확인.
* **비즈니스 로깅:** 어떤 컨트롤러의 메소드가 실행되었는지 등 상세 비즈니스 로직 관련 로그 기록.
* **데이터 가공:** 컨트롤러로 전달되는 `Model` 객체에 공통 데이터 추가.
* **성능 측정:** 특정 컨트롤러의 실행 시간 측정.

-----

### 4\. AOP 관점에서의 비교

AOP(Aspect-Oriented Programming, 관점 지향 프로그래밍)는 **핵심 비즈니스 로직과 공통 기능(부가 기능)을 분리**하여 코드의 모듈성을 높이는 프로그래밍 패러다임이다.

이러한 관점에서 필터와 인터셉터는 웹 애플리케이션의 대표적인 \*\*횡단 관심사(Cross-Cutting Concerns)\*\*를 처리하는 기술로 볼 수 있다. 둘 다 여러 컨트롤러나 비즈니스 로직에 흩어져 있는 공통 코드를 모아주는 역할을 수행한다.

-----

### 5\. 실무 적용 사례

필터와 인터셉터는 처리하는 횡단 관심사의 '계층'에 따라 구분되어 사용된다.

#### 5.1. 필터(Filter) 적용 사례: 인프라 레벨의 횡단 관심사

필터는 \*\*웹 애플리케이션의 가장 바깥쪽(서블릿 컨테이너 레벨)\*\*에서 동작하므로, 스프링과 독립적인 **범용적이고 인프라에 가까운 횡단 관심사** 처리에 적합하다.

* **인코딩 설정 (Character Encoding)**
    * **횡단 관심사:** 모든 HTTP 요청과 응답의 문자 인코딩을 `UTF-8` 등으로 통일하는 작업.
    * **AOP 적용:** `Encoding Filter`를 구현하여 애플리케이션에 들어오는 모든 요청(`/*`)에 일괄 적용한다.
* **전역적 보안 처리 (Global Security)**
    * **횡단 관심사:** 모든 요청에 대한 XSS 공격 방지, 특정 IP 대역 접근 차단 등.
    * **AOP 적용:** `Security Filter`를 통해 핵심 로직과 분리하여 보안 정책을 일괄 적용한다. (예: Spring Security의 `DelegatingFilterProxy`)
* **요청/응답 데이터 압축 (Data Compression)**
    * **횡단 관심사:** 클라이언트로 전송하는 응답 데이터를 GZIP 등으로 압축하는 작업.
    * **AOP 적용:** `Compression Filter`를 통해 응답 직전 데이터를 압축하여 네트워크 트래픽을 절감한다.

-----

#### 5.2. 인터셉터(Interceptor) 적용 사례: 비즈니스 레벨의 횡단 관심사

인터셉터는 \*\*스프링 컨테이너 내부(컨트롤러 진입 전)\*\*에서 동작하므로, **애플리케이션의 비즈니스 로직과 밀접하게 관련된 횡단 관심사** 처리에 유용하다.

* **사용자 인증 및 권한 검사 (Authentication & Authorization)**
    * **횡단 관심사:** `/mypage/*`, `/admin/*` 등 특정 URL 패턴 접근 시 사용자의 로그인 상태나 관리자 권한을 확인하는 작업.
    * **AOP 적용:** `LoginCheckInterceptor`를 구현하여 컨트롤러 실행 전 세션이나 토큰을 검사함으로써, 인증/인가 코드를 비즈니스 로직과 분리한다.
* **상세 로깅 및 성능 측정 (Detailed Logging & Performance Monitoring)**
    * **횡단 관심사:** 특정 컨트롤러 메소드의 호출 시점, 실행 시간, 파라미터 등을 기록하는 작업.
    * **AOP 적용:** `LoggingInterceptor`의 `preHandle` (시작 시간)과 `postHandle` (종료 시간)을 활용해 성능을 측정하고 로그를 남긴다.
* **공통 데이터 주입 (Common Data Injection)**
    * **횡단 관심사:** 모든 페이지 상단에 표시될 사용자 닉네임, 알림 개수 등 공통 정보를 모델에 담는 작업.
    * **AOP 적용:** `CommonDataInterceptor`의 `postHandle` 시점에서 `ModelAndView` 객체에 공통 데이터를 추가한다.

-----

### 6\. 요약

필터는 스프링과 무관하게 \*\*전역적인 요청/응답(인코딩, 보안, 압축 등)\*\*을 처리하는 데 사용된다. 반면 인터셉터는 \*\*스프링과 연관된 비즈니스 로직(인증, 상세 로깅, DB 접근)\*\*을 컨트롤러 진입 전후에 처리하는 데 특화되어 있다. 따라서 인증되지 않은 사용자의 요청을 DB 접근 전에 차단하려면 필터가 더 효율적이며, 비즈니스 로직과 연계된 공통 기능은 인터셉터가 적합하다.
